# 🤖 LLM 編碼行為指南 (LLM Coding Guidelines)

> **核心精神**：謹慎優先於速度。對於瑣碎任務，請使用判斷力靈活運用。

來源：`CLAUDE.md` — 適用於減少 LLM 常見編碼錯誤的行為準則。

---

## 原則一：先思考，再動手 (Think Before Coding)

> 不要假設。不要隱藏疑惑。主動揭露取捨。

**實作前必須做到：**

- **明確陳述假設**：如果不確定，就直接問。
- **列出所有解讀**：若有多種解讀方式，全部列出，不要靜默挑選其中一個。
- **提出更簡單的解法**：若存在更簡單的方式，說出來。必要時，推回（push back）需求。
- **卡住就停下**：說明哪裡不清楚，然後發問。

**反模式（應避免）：**
- 假設使用者的意圖，悄悄選一個實作方向
- 對模糊的需求直接動工，事後再問「你說的是這個嗎？」

---

## 原則二：簡單優先 (Simplicity First)

> 最少的程式碼解決問題。不寫任何推測性的程式碼。

**禁止事項：**

| 行為 | 說明 |
| ---- | ---- |
| 超範圍功能 | 不加使用者沒要求的功能 |
| 過度抽象 | 不為單次使用的程式碼建立抽象層 |
| 假性彈性 | 不加沒被要求的「彈性」或「可配置性」 |
| 防禦性過度 | 不處理不可能發生的例外情境 |

**自我檢查問題**：
> 「資深工程師會覺得這太複雜嗎？」— 若答案是「是」，就簡化它。

如果你寫了 200 行，但 50 行就夠了，**重寫它**。

---

## 原則三：外科手術式修改 (Surgical Changes)

> 只動必要的地方。只清理自己製造的問題。

**修改現有程式碼時：**

- ❌ 不要「順便改善」無關的程式碼、註解或格式
- ❌ 不要重構沒有壞掉的東西
- ✅ 維持現有風格，即使你有不同偏好

**清理孤兒 (Orphan Cleanup) 規則：**

| 情境 | 應對措施 |
| ---- | -------- |
| 你的修改造成的孤兒 import/變數/函式 | ✅ 移除 |
| 原本就存在的廢棄程式碼 | ❌ 不刪（除非被明確要求）|
| 發現不相關的死碼 | ✅ 提及它，但不刪除 |

**檢驗標準**：
> 每一行改動都必須能直接追溯到使用者的需求。

---

## 原則四：目標驅動執行 (Goal-Driven Execution)

> 定義成功標準。循環直到驗證完成。

**將模糊任務轉為可驗證目標：**

| 模糊描述 | 可驗證目標 |
| -------- | ---------- |
| 「加入驗證」 | 先為無效輸入寫測試，再讓測試通過 |
| 「修復 Bug」 | 先寫重現測試，再讓測試通過 |
| 「重構 X」 | 確保重構前後測試都通過 |

**多步驟任務必須列計畫：**

```
1. [步驟] → 驗證: [如何確認]
2. [步驟] → 驗證: [如何確認]
3. [步驟] → 驗證: [如何確認]
```

> 強力的成功標準讓你能獨立循環驗證。弱標準（「讓它能動就好」）需要不斷澄清。

---

## ✅ 指南是否有效的判斷指標

- Diff 中的**不必要變更減少**了
- 因過度複雜而被要求**重寫的情況減少**了
- **澄清問題在實作前提出**，而非出錯後才問
