# Advisor Workflow

> 規劃 -> 專業分析 (平行) -> 戰略彙整 -> 第二意見 -> 驗證與紀錄

## 🎯 模式選擇

| 模式         | 用法               | 說明                                |
| ------------ | ------------------ | ----------------------------------- |
| **完整模式** | `/advisor`         | 執行 6 步驟完整流程（含規劃與驗證） |
| **快速模式** | `/advisor --quick` | 僅收集數據 + PM 檢視 + 最小驗證     |

## 0. 規劃階段 (Plan First)

> ⚠️ 任何非瑣碎任務（3+ 步驟或涉及架構決策）都必須先規劃。

### 明確任務目標

在開始分析前，先回答以下問題：

1. **今天要解決什麼問題？** (一句話描述)
2. **預期產出是什麼？** (具體可交付物)
3. **成功標準是什麼？** (如何判斷「完成」)

將答案寫入 `tasks/todo.md` 作為本次同步的 checkpoint。

## 2. 啟動層級式分析流程

### 第一步：專業視角分析 (平行執行)

> 💡 三位專家的分析互相獨立，應盡可能平行執行以提升效率。

請根據上述數據，分別使用以下模板產出分析報告：

- 使用 `[[web-pm-prompt]]` (產品與進度視角)
- 使用 `[[cto-prompt]]` (技術架構與效能視角)
- 使用 `[[designer-prompt]]` (UI/UX 與視覺視角)

每位專家必須：

- 遵循「Simplicity First」原則：建議應盡可能簡單
- 遵循「Minimal Impact」原則：變更應只觸及必要的部分
- 找出根本原因，不做臨時修補 (No Laziness)

> 🚨 **架構遷移判斷**：如果任何專家的建議涉及**資料源切換、框架遷移、或底層重寫**，
> CTO 必須啟動 `cto-prompt.md` 中的「架構變更影響評估」模板，填寫影響範圍清單後再往下進行。

### 第二步：超級角色彙整 + 優雅度檢查

將上述報告交給 CSO 進行初步戰略定義：

- 使用 `[[cso-prompt]]` (首席策略官)
- CSO 必須額外回答：**「有沒有更優雅的方案？」**
- 對非瑣碎的建議，暫停並反思是否過度工程化

### 第三步：第二意見審視 (Devil's Advocate)

將 CSO 的決策交給 CIO/CRO 進行挑戰與審視：

- 使用 `[[cio-prompt]]` 或 `[[cro-prompt]]`
- 專家會從風險、盲點、替代方案角度提出質疑
- 必須回答：**「一位資深工程師會批准這個決策嗎？」**

### 第四步：最終共識

綜合以上視點，形成最終行動方案與 Backlog 調整建議。

## 3. 驗證與紀錄 (Verify Before Done)

> ⚠️ 永遠不要在沒有驗證的情況下標記任務完成。

### 品質自查清單

- [ ] 最終建議是否回答了 Step 0 的目標問題？
- [ ] 行動項目是否具體、可執行？
- [ ] 是否遺漏任何重大風險或盲點？
- [ ] 建議是否符合「Simplicity First」原則？

### 更新追蹤文件

1. 在 `tasks/todo.md` 中標記已完成的項目
2. 如果本次分析中有任何「修正」或「教訓」，記錄到 `tasks/lessons.md`
